# `tests`

This directory contains the plugin's pytest test suite.

## Files And Ownership

- `conftest.py`: shared fixtures and compatibility stubs for Django/NetBox/requests so tests can run without a full NetBox install. Includes `StreamingHttpResponse` stubs and mock request helpers.
- `django_stubs.py`: the single source of the `django.db` / `django.utils.crypto` stub pair that every path-loader of `views/backend_sync.py` installs. **Six** independent loaders exist ‚Äî `conftest.py::load_plugin_module()`, `test_jobs.py::load_real_backend_sync()`, and the module loaders in `test_multi_endpoint_scoping.py`, `test_preflight_diagnosis.py`, `test_backend_sync_placement.py`, and `test_endpoint_enabled_guards.py` ‚Äî and each must install the pair *itself* rather than relying on a predecessor: `test_hardware_discovery_custom_fields_migration.py` leaks a **partial** `django.db` into `sys.modules` permanently, so whichever loader runs after it would otherwise import a `django.db` with no `DatabaseError`. Exports `TEST_SECRET_KEY`, `DatabaseError`, a faithful `salted_hmac()`, `django_stub_modules(*, models_module=None)`, and `install_django_stubs(monkeypatch, *, models_module=None)`. `salted_hmac` is **implemented**, not stubbed to a constant ‚Äî a constant would make every credential fingerprint compare equal, which is the exact opposite of what the fingerprint detects, and every fingerprint test would pass while testing nothing. Only those two dotted names are needed; `backend_sync.py` touches nothing else in Django at import time.
- `test_api_cluster.py`, `test_api_netbox_integration.py`, `test_api_source_contracts.py`: API layer, cluster serializer, and serializer contract checks. `test_api_source_contracts.py` also covers the ten non-model `APIView` classes (`HomeAPIView`, `DashboardAPIView`, resource views, `ScheduleSyncAPIView`, `BackendLogsAPIView`): class existence, base class, permission class assignment, HTTP methods, URL registration, API root keys, and serializer field contracts.
- `test_backend_integration.py`, `test_backend_logs_view.py`, `test_job_stream.py`, `test_run_sync_stream.py`, `test_sse_contracts.py`: backend proxy, log view, stream, and SSE behavior. `test_run_sync_stream.py`'s **redaction** section pins that credentials cannot reach the NetBox job log through the stream error path: a failed `complete` frame echoing the pushed endpoint body is redacted **including its raw `response` mirror** (the `ok is False` branch returns `last_complete.model_dump()` verbatim, so redacting `detail` alone is not enough), `on_frame` receives redacted frame data, and `_redacted_mapping()` stays fail-closed when redaction returns a non-mapping. Three counter-tests keep redaction from becoming lossy: the **success** payload's sync counters are untouched, the rejected field's *name* and `msg` survive so the error stays diagnosable, and the two substring markers downstream code branches on (`"init_ok"` in `sync_stages.py`, the Postgres `remaining connection slots ‚Ä¶` phrase in `sync_types.py`) still match end-to-end. The fixture `sys.modules.pop`s `netbox_proxbox.views.error_utils` so the **real** redactor loads rather than a stub.
- `test_cards.py`, `test_dashboard.py`, `test_keepalive_status.py`: dashboard/card hydration and service status checks.
- `test_cli_contracts.py`, `test_form_and_helper_source_contracts.py`, `test_frontend_contracts.py`: CLI, form/helper, and DOM/template/view contract coverage.
- `test_jobs.py`, `test_job_cancel.py`, `test_vm_sync_now_view.py`: job queueing, cancellation, and targeted VM sync behavior. `test_jobs.py` also owns the sync-job **preflight** behavior tests ‚Äî that a confirmed-empty backend or a missing/unusable FastAPI backend blocks the run before any stage, that an ambiguous or already-populated backend does not, that a NetBox with **zero enabled `NetBoxEndpoint` rows** blocks unconditionally ‚Äî with stale rows deliberately present on the backend, and the test asserting `list_backend_netbox_endpoints()` is never even called ‚Äî that **zero enabled `ProxmoxEndpoint` rows** produce a fail-loud `endpoint-scope` record instead of an unscoped all-backend stage request (again with stale backend rows present, asserting the SSE transport is never reached), that the happy path emits no hint (a spurious one would misattribute unrelated stage failures), that the selected `fastapi_endpoint_id` reaches both the context resolver and key registration (with two enabled backends, checking the wrong one fails a working sync), that endpoint-scope failures fail the job with a message distinguishing "No sync stage ran" from "N Proxmox endpoint(s) were skipped", that an unresolvable endpoint whose every selected stage is sync-mode-disabled is recorded as a *skip* rather than failing the job, that the selected backend pk reaches all four pre-SSE service passes (cluster/node, firewall, datacenter, VM templates), and that `PreflightResult` does not share a mutable default. Three tests cover the push budget: `PREFLIGHT_ENDPOINT_PUSH_BUDGET` cuts the loop short while still emitting a warning phase per skipped endpoint, the budget **never** skips an endpoint the backend does not already hold (skipping it would leave it without a wire id and fail the job outright), and `PREFLIGHT_ENDPOINT_PUSH_HARD_CEILING` skips even unregistered endpoints. Those tests drive a fake `time.monotonic` from the push stub itself rather than sleeping. A fourth budget test, `test_preflight_budget_never_skips_an_endpoint_whose_credentials_rotated`, pins the Proxmox credential fingerprint (migration 0074) at the job level: with every backend row held and current on the wire, an endpoint whose secret rotated after its fingerprint was recorded ‚Äî and one whose fingerprint was never recorded ‚Äî are both pushed despite an exhausted budget, because `ProxmoxEndpointPublic` withholds the credential fields and only the local receipt can tell a no-op refresh from a push that delivers a new secret (`_arrange_slow_push` records the held rows' fingerprints through the **real** helper for exactly this reason). The recording end has its own Proxmox twins ‚Äî `test_recording_a_pushed_proxmox_fingerprint_updates_the_row_without_saving_it` and `‚Ä¶_survives_an_unapplied_migration` ‚Äî mirroring the NetBox pair (write through `queryset.update()`, never `save()`; a missing 0074 column degrades to always-push instead of failing the push), plus `test_proxmox_and_netbox_credential_fingerprints_never_compare_equal`, which feeds both fingerprint functions identical material and pins that the distinct salts keep the two namespaces disjoint. `test_run_scopes_firewall_and_datacenter_to_the_runs_endpoints` pins that `run()` forwards `endpoint_ids_to_sync` to the firewall and datacenter passes, which used to build their own all-enabled scope and ignore the job's selection. They also share `_preflight_proxmox_endpoint(pk)` / `_arrange_slow_push(..., held=‚Ä¶)`, which build **complete, current** backend rows rather than bare `{"name": ‚Ä¶}` stubs ‚Äî `backend_holds_proxmox_endpoint()` now means "held *and* current", so a row missing `port`/`username`/`access_methods`/`verify_ssl` reads as drifted and the budget correctly refuses to skip it, which is not the behavior those three tests are about. A further group pins the **backend-pin round trip** that keeps a *replayed* job on the same proxbox-api: `enqueue()` persists `fastapi_endpoint_id` into `job.data['proxbox_sync']['params']` (coercing `"7"` ‚Üí `7` and dropping an unusable value entirely rather than storing a half-parsed pk), still forwards the kwarg to `run()`, omits the key when unpinned, and `proxbox_sync_params_from_job()` reads it back on **both** return paths ‚Äî including the legacy targeted-VM name inference, which rebuilds the params from scratch. `_coerce_fastapi_endpoint_id` is parametrized directly, `True`/`False` included (a `bool` is an `int` in Python and must not become endpoint 1). Two further tests pin that the coercion happens **before** `super().enqueue()` and **again** at the top of `run()`, which is what keeps the queued RQ kwargs and `job.data` from disagreeing: `test_enqueue_never_queues_a_pin_it_refused_to_store` asserts an unusable pin appears in *neither*, and `test_run_recoerces_an_unusable_backend_pin_before_any_lookup` asserts `run()` normalises it before the preflight sees it (direct callers, and RQ payloads queued by a release predating the `enqueue()` coercion, still hand it a raw value). `True` is the sharp case both ways ‚Äî it survives every guard and Django resolves `pk=True` against primary key 1. A further group pins that **a stored backend row is only trusted once it is identified**: a failed push plus a row whose **resolved connection target** ‚Äî `(domain or ip_address)` plus `port` ‚Äî equals this NetBox's warns "may be stale" and continues, while a row with the *same name* but a different target, the right host on a different port, a *different* domain sharing our configured IP, a blank domain at our IP while we declare one, a row reporting no port at all, or nothing but the synthetic `127.0.0.1` push fallback all block with "do not point at this NetBox". Matching by name would pass the first ‚Äî the backend NetBox endpoint is a singleton updated by position and its payload name is free text. **That positional singleton is itself asserted**: `test_preflight_blocks_when_only_a_later_backend_row_is_ours` feeds a two-row listing whose *second* row is ours and pins that the run blocks anyway, with no "may ‚Ä¶ stale" warning, because a match at index 1 vouches for a row proxbox-api is not dialling while the stale row at index 0 drives the sync. `test_preflight_continues_when_backend_already_holds_an_endpoint` therefore feeds the *single* row it always claimed to. Two tests deliberately feed the *same* stored row to opposite verdicts: reached-by-address blocks a domain-configured NetBox and continues an IP-only one, because which field the backend dials is what decides identity. Two more pin fix N, that **"unknown" must not read as "ours"**: a failed push whose backend listing *also* failed blocks, unless some other enabled row pushed successfully (that push already wrote this NetBox's credentials into the singleton, so the unreadable listing has nothing left to add). **These tests call the real `backend_holds_netbox_endpoint()` / `backend_holds_proxmox_endpoint()`, installed on the fixture's fake module by `load_real_backend_sync()` ‚Äî never a hand-written mirror.** That is not a style preference. The fixture used to re-implement the predicate "faithfully", and when the real helper was found to match two different domain-only NetBoxes against each other through the loopback fallback, every test still passed, because they were asserting against the copy. The defect survived six adversarial review rounds that way. `load_real_backend_sync()` path-loads `views/backend_sync.py` once against throwaway stubs (saving and restoring the eight `sys.modules` entries in a `try/finally`) and takes only the pure predicates; everything that performs HTTP is stubbed separately, so nothing there can reach the network. `test_backend_holds_netbox_endpoint_identity_contract` then states the contract directly as a 21-case table against that same real function, in five groups ‚Äî positive identification, the loopback trap, a different resolved connection target, the port, and the fail-closed inputs ‚Äî because the `run()`-level tests only exercise one verdict each. The fail-closed group is where the **positional singleton** is pinned: a single junk entry is `False` without raising, and *any* listing longer than one row is `False` too ‚Äî including two of **our own** rows, and junk followed by ours ‚Äî because the push overwrites entry `[0]` and entry `[0]` is what the backend dials, so a longer list means nothing in the response says which row wins. The table is what pins the two directions a field-by-field comparison gets wrong: a stored row blank on `domain` at our IP, or naming *another* domain at our IP, must **not** match (both dial a different service at that address), while our own row whose stored IP has since drifted **must** still match, because a row with a domain never dials its address at all. Its rows are built by `_stored_row()`, which supplies a *current* `verify_ssl`/`token_version` by default, because identity and **currency** are separate questions and that table is only asking the first one. `test_backend_holds_netbox_endpoint_currency_contract` asks the second as its own 10-case table: a `verify_ssl` or `token_version` that drifted in **either** direction is refused, whitespace-only differences are not, an *absent* `verify_ssl` or `token_version` reads as drifted (a `False` here blocks the run, so unknown must fail closed), and **currency is only ever asked of the row the backend dials** ‚Äî a current duplicate sitting *behind* a drifted row does not rescue it, and neither does a current row belonging to a *different* NetBox, both now twice over since a listing longer than the positional singleton is refused before currency is asked at all. Three preflight-level tests take it end to end ‚Äî a superseded TLS posture, a superseded token scheme, and a row omitting a mandatory trust field all block the run rather than warning and continuing. Four more pin the **secret** the currency check structurally cannot see, because `NetBoxEndpointResponse` withholds `token`/`token_key`: `test_preflight_blocks_when_the_stored_row_predates_an_in_place_token_rotation` (identity says "ours" and currency says "current", both correctly ‚Äî and the run would still proceed on a revoked token, so the locally recorded fingerprint of the last **successful** push is the only thing left that can tell), `test_preflight_blocks_when_no_push_ever_recorded_a_credential_fingerprint` (an *empty* fingerprint reads as drifted, because unknown must fail closed; one successful push clears it permanently), and two on the writer itself ‚Äî `test_recording_a_pushed_fingerprint_updates_the_row_without_saving_it` asserts it goes through `type(endpoint).objects.filter(pk=pk).update(...)` and **never** `save()`, since `NetBoxEndpoint`'s `post_save` handler re-pushes to the backend and saving from inside a push would recurse, while `test_recording_a_pushed_fingerprint_survives_an_unapplied_migration` feeds it the `ProgrammingError` an install without migration `0073` raises and asserts the push still succeeds with the fingerprint left empty (i.e. fail-closed, not fail-loud). `_arrange_preflight(..., netbox_fingerprint="current"|"stale"|"absent")` selects the three states, computing `"current"`/`"stale"` through the **real** `netbox_endpoint_credential_fingerprint()` rather than a literal ‚Äî `salted_hmac` is implemented faithfully in `tests/django_stubs.py` for exactly this reason, since stubbing it to a constant would make every fingerprint compare equal and silently delete the test. Four more pin the **selected-object batch path**: it blocks with zero enabled `NetBoxEndpoint` rows and never calls `_run_batch_selected_sync` (it used to return *before* the preflight entirely), it forwards the job's `fastapi_endpoint_id` into that call, `_run_batch_selected_sync` itself passes the id to every per-object `sync_individual_with_dependencies`, and ‚Äî via `_arrange_batch_storage_sync()`, shared by the last two so both assert the **whole** call dict rather than one key ‚Äî it passes `netbox_branch_schema_id` as a keyword and not only inside `query_params`. That last one is not cosmetic: `_sync_dependency()` rebuilds its params from `_CONTEXT_KEYS`, which omits the branch key, so a query-param-only schema id would write the selected object to the branch and every dependency resolved off it to **main**. Those two drive `sync_stages` directly, so they need the `proxbox_sync_job_module` fixture for the `netbox.*` stubs **and** `monkeypatch.setitem` stubs for `virtualization`, `virtualization.models`, `netbox_proxbox.models`, and `netbox_proxbox.services.individual_sync` ‚Äî `_run_batch_selected_sync` resolves all four lazily, at call time, so `conftest.py`'s module-level stubs do not apply. Four further tests pin that the batch path also resolves the **Proxmox** scope, which fails in the opposite direction from the branch schema: `_run_batch_selected_sync` forwards `proxmox_wire_endpoint_ids` to dependency syncs as both a query param and an argument, the full `run()` asserts the resolved scope (`"1"`) reaches `_run_batch_selected_sync`, and ‚Äî the two the review round mandated ‚Äî a run with **no enabled Proxmox endpoint** and a run whose enabled endpoints **resolve to no backend id** both raise `ProxboxPreflightError` with `batch_calls == []`, i.e. proxbox-api is never called. Both use `_arrange_blocked_batch()`, which deliberately leaves *stale* rows on the backend listing: those rows are exactly what makes an unscoped request dangerous, so the run must stop anyway. `test_batch_wire_endpoint_scope_narrows_rather_than_failing_on_partial_drift` pins the design decision that one drifted endpoint narrows the scope (`("11", ["2"], None, {"1": "11"})`) instead of failing the run. Five further tests pin the **per-object pinning** that narrowing alone does not give you, all built on `_arrange_batch_duplicate_identifier_sync()`: two `ProxmoxStorage` rows that are *indistinguishable to proxbox-api* ‚Äî both `local-lvm` on a `cluster01` ‚Äî differing only in `cluster_id`, so the two requests they produce are byte-identical apart from the endpoint scope. That is the whole point: Proxmox identifiers are unique per endpoint, not across the estate, so with the job-wide scope both endpoints can answer and one row silently gets the other estate's storage. The five cover each branch of the **four-state** owner resolution ‚Äî both objects pinned to their **own** wire id rather than the job-wide `"11,22"`; an object whose owner resolves but is **absent** from the map refused with status `424` and *no* backend call appended to `calls`; an object with no reflected `ProxmoxCluster` read as *unknown* and falling back to the job scope **only where that scope names one endpoint** (`"11"`, the first-ever-sync case, which refusing would make unsyncable forever ‚Äî and where falling back and pinning are the same request, so nothing is being guessed); the *same* unknown object across a **two**-endpoint scope refused with `424` saying the run "spans 2 Proxmox endpoints", since there the fallback is a guess made with identifiers that are unique only per endpoint, and its error names the single-endpoint retry that makes it recoverable; and a cluster claimed by **two** endpoints ‚Äî *ambiguous*, which is **not** the same state as unknown ‚Äî refused with `424` naming every claimant, because two endpoints reflecting one cluster is proof the duplicated namespace exists in this estate, so widening back to the job-wide scope would ask both and keep whichever answered (the exact defect being fixed) while picking one would be a guess about where the object lives. Collapsing ambiguous into unknown restores that widening, and the ambiguity test is what fails when someone does. `_proxmox_cluster_objects()` stubs the `.filter(‚Ä¶).exclude(‚Ä¶).values_list(‚Ä¶).distinct()` chain `_owner_endpoint_pks_by_cluster_id()` issues. `test_batch_selected_sync_fails_the_job_when_selected_objects_failed` pins the enclosing consequence: a `batch_result["failed"] > 0` raises rather than finishing **completed**, since every object in a selected-object run was named by an operator and there is no "partial success" reading of a list somebody typed out ‚Äî and the per-object `424` refusals above would otherwise be visible only in `job.data`. It asserts both orderings that make the failure useful: `job.data` is saved *before* the raise (so the failed row still carries the per-object status and error) and the raise happens *before* the branch merge, with branching deliberately **enabled** so "the merge did not run" is an assertion about ordering rather than about branching being off. A separate 8-case parametrized table pins `_batch_object_core_cluster_id()` ‚Äî that all five batch types converge on a core `virtualization.Cluster` id, that a backup prefers its *storage's* cluster (the one its own sync params are built from) while a snapshot prefers its VM's, and that an unresolvable object or unknown type returns `None` rather than guessing. **Patch `effective_sync_modes_for_endpoint` on both `jobs` and `jobs.sync_stages`** ‚Äî `jobs._sync_stage_settings()` re-copies jobs.py's own reference onto `sync_stages` on every run, silently reverting a patch applied to only one of them.
- `test_preflight_diagnosis.py`: regression suite for netbox-proxbox issue #624 ‚Äî a fresh install whose sync died on stage `devices` with an opaque `Error ensuring Proxbox tag`. Pins the three behaviors that turn that into a diagnosable failure: `list_backend_netbox_endpoints()`'s three-way `(rows, None)` / `(None, error)` contract, the named cold-start timeout budgets in `services/backend_auth.py`, and `_is_retryable_stage_failure()` plus `preflight_hint` attribution in `sync_stages.py`. Reads `backend_auth.py`'s constants with `ast.literal_eval` rather than importing it (the module pulls in Django at import time) and asserts the key-registration call sites use the *named* constants via an AST walk, so a future inline literal cannot silently reintroduce a too-tight budget. `TestRetryableStageFailure` additionally pins the two marker-matching mistakes the classifier must not make: a **false positive** (HTTP 400 `"timeout must be between 1 and 300"` is a validation rejection, not a transport failure ‚Äî which is why no bare single-word marker may be added) and a **false negative** (a transport cause nested in a FastAPI `detail` **list** must still retry), plus the invariant that the echoed `input` field is never scanned and that cyclic/over-deep payloads terminate on `_CAUSE_RECURSION_LIMIT`. `test_real_transport_failure_texts_are_matched` parametrizes the *verbatim* strings the transport layer emits (nginx's hyphenated `504 Gateway Time-out` body, urllib3's `HTTPConnectionPool ‚Ä¶ Read timed out`, `[Errno 111] Connection refused`, httpx's `Server disconnected without sending a response.`, glibc `EAI_AGAIN`/`EAI_NONAME` and BSD resolver errors, and an `ssl` `EOF occurred in violation of protocol`) so a marker can never be reduced to a shape that matches only the paraphrase ‚Äî four of those eight were genuine misses before this test existed.
- `test_endpoint_scope.py`: behavior tests for `services/endpoint_scope.py::enabled_backend_endpoint_scope`, loaded by file path against a capturing manager and resolver stub. Pins the `endpoint_ids` narrowing contract the firewall/datacenter pre-SSE passes rely on: `None` keeps the historic all-enabled scope (no pk filter), a non-empty selection reaches the ORM as `pk__in` beside `enabled=True` and only the surviving endpoints are resolved to backend ids, and an **empty** selection ‚Äî or one matching no enabled endpoint ‚Äî returns `(None, {}, None)` before the ORM and the backend are consulted, because falling through would build the widest request proxbox-api accepts exactly when the caller asked for the narrowest.
- `test_multi_endpoint_scoping.py`: behavior tests for `views/backend_sync.py`'s wire-id resolution, loaded by file path (`_load_backend_sync_module`) against hand-built stubs ‚Äî no NetBox bootstrap and no HTTP. Pins `proxmox_backend_name()`'s `"<name> (nb:<pk>)"` shape, the disabled-endpoint short circuits (both `sync_proxmox_endpoint_to_backend()` and `resolve_backend_endpoint_id()` return before any `requests.get`), the unregistered and list-error paths, and the batch omission behavior. Its largest group pins that **a located row must still be confirmed to dial the same service**: a row pointing at another host, one on a different port, one reporting no port at all, one matching only the *unused* stored address, and an endpoint that resolves no host itself are all refused, while an address change *under a configured domain* is correctly ignored ‚Äî because a row with a domain never dials its address. `test_backend_holds_proxmox_endpoint_requires_the_row_to_be_current` states the soft-budget predicate as a table (current row held; retargeted / different port / user / transport / TLS / another endpoint all still pushed; empty **and** `None` both not-held, since a failed listing is *unknown* and unknown must never skip a push). "Current" now includes the **credentials**, which the backend row cannot report: the held case records a fingerprint through the real `proxmox_endpoint_credential_fingerprint()` first, and two further cases pin that a secret rotated in place after that recording, and a never-recorded fingerprint, both read as "push again" rather than "held". The loader's `disabled_endpoint_detail` stub takes `(endpoint, **kwargs)` ‚Äî the real function's `kind=` / `action=` are keyword-only and the call site passes them, so a bare `lambda endpoint:` raises `TypeError` from inside the code under test.

## Notes

- Tests rely heavily on compatibility stubs in `conftest.py` rather than a live NetBox database.
- When changing view, job-enqueue, or template contracts, update both the runtime code and those stubs.
- `test_frontend_contracts.py` guards against accidental regressions in public-facing attribute names and JS function signatures.
- **Wire-contract tests:** `test_sse_schema_mirror.py` validates local REST/SSE schema mirrors against `contracts/proxbox_api_sse_schema.json`. It must not import or install `proxbox-api`; the plugin and backend communicate over HTTP, not direct Python calls.
- **AST-based source contracts** (e.g. `test_version.py`, `test_signals.py`, `test_services_backend_context.py`, `test_proxmox_endpoint_settings_view.py`, `test_views_vm_config.py`, `test_views_storage.py`) parse the relevant module with `ast` and never bootstrap Django. Use this pattern when the runtime cost of starting NetBox would dominate the test value. When such a test needs a module's constant values, read them with `ast.literal_eval` ‚Äî never `eval()`, which the workspace security hook rejects outright.

### Stub-loader constraints

Tests that load a plugin module by file path (`spec_from_file_location` +
`monkeypatch.setitem(sys.modules, ...)`) run it against hand-built stub modules
rather than real NetBox. Two failure modes are easy to hit and hard to read:

- **No `@dataclasses.dataclass` in a module that uses `from __future__ import
  annotations`.** Under Python 3.14, `dataclasses._is_type` resolves string
  annotations through `sys.modules[cls.__module__]`, which is `None` for a
  path-loaded module the loader never registered ‚Äî so the decorator raises
  during import, far from the actual test. `jobs.py`'s `PreflightResult` is a
  plain class with `__slots__` for exactly this reason.
- **Stub the whole `netbox_proxbox.services` package, not individual
  submodules.** `jobs.py` imports `netbox_proxbox.services.branch_lifecycle`
  lazily inside `run()`, guarded only by `except ModuleNotFoundError`. A partial
  stub lets the real `services/__init__.py` execute, which drags in
  `backend_proxy.py` and raises a chain of `ImportError`s ‚Äî a *different*
  exception class, so the guard does not catch it and the test fails on an
  unrelated import. `monkeypatch.setitem(sys.modules,
  "netbox_proxbox.services", services_mod)` is the established idiom; see the
  `run()`-level tests in `test_jobs.py`.
- **A new `from netbox_proxbox.sync_params import ‚Ä¶` name must be added to the
  stub in `test_service_monitoring_scheduler.py`.** That test installs a
  hand-built `netbox_proxbox.sync_params` module listing each symbol `jobs.py`
  imports; a name missing from that list fails at *import* time with
  `ImportError: cannot import name ‚Ä¶ (unknown location)`, in a test that has
  nothing to do with the change.
- **The `proxbox_sync_job_module` fixture models a *normal* deployment: one
  enabled Proxmox endpoint that resolves to a backend wire id.** Its
  `_EndpointManager.filter()` synthesizes rows on demand ‚Äî per requested pk for a
  `pk__in`/`pk` lookup, and a single default endpoint (pk 1) for an *unfiltered*
  `filter(enabled=True)` ‚Äî and the fixture installs a default
  `sync_stages._resolve_wire_endpoint_ids` that maps each scoped pk to itself.
  Both defaults are load-bearing: without them every `run()`-level test lands in
  the "no enabled Proxmox endpoint" / "never resolved to a backend id" fail-loud
  branches, exercising a disabled-estate path none of them meant to test, and the
  failure surfaces as ~19 unrelated tests breaking at once. A test that wants the
  empty estate must override `models.ProxmoxEndpoint` itself (see
  `test_proxmox_endpoint_scopes_returns_no_scope_when_every_endpoint_disabled`);
  a test that wants a resolution failure re-patches `_resolve_wire_endpoint_ids`,
  which wins because test bodies run after the fixture.
- **Load a security predicate, never mirror it.** When a stubbed module needs a
  helper that decides *whether an operation may proceed* ‚Äî as opposed to one
  that merely performs I/O ‚Äî install the real function via a path-loader instead
  of re-implementing it on the fake module. `test_jobs.py`'s
  `load_real_backend_sync()` is the pattern: stub `netbox_proxbox`,
  `.models`, `.utils`, `.services`, `.services.endpoint_enabled`, `.views`, and
  `.views.error_utils`, `spec_from_file_location` the target, then restore every
  saved `sys.modules` entry in a `finally`. A mirror silently stops testing the
  thing it names the moment the real code changes, and the tests keep passing
  while it does.
- **Coverage:** `pyproject.toml` defines `[tool.coverage.run]` with `source = ["netbox_proxbox"]` and `branch = true`, plus `[tool.coverage.report]` with `exclude_lines` for `TYPE_CHECKING` and `pragma: no cover` markers. CI invokes `pytest --cov=netbox_proxbox`.
